Advanced Milvus Search Techniques: Range, Grouping, and Hybrid Search

Mar 02, 2026
8 minute read
简体中文

Coverage: Range search, grouping search, hybrid search (multi-vector + reranking) Official Documentation: Range Search | Grouping Search | Hybrid Search

I. Range Search

1.1 Core Concepts

Normal ANN search returns “K most similar” results, but doesn’t limit the similarity range. Range search uses radius and range_filter parameters to define a ring-shaped similarity interval, returning only results that fall within the interval.

Using query vector as center, draw two concentric circles:

  range_filter (inner circle)── Exclude overly similar results (deduplication scenarios)
  radius (outer circle)──────── Exclude overly dissimilar results

  Only return vectors falling within the ring-shaped area

Why do we need the inner circle (range_filter)?

  • Deduplication scenarios: Exclude results almost identical to query vector (might be the query vector itself)
  • Recommendation scenarios: Exclude highly similar content users have already seen, recommend “related but non-repetitive” content
  • If inner circle not needed, set range_filter to 0 (L2/JACCARD/HAMMING) or leave unset

1.2 Parameter Setting Rules (Important, easy to confuse)

Different distance metrics have opposite parameter relationships:

metric_type Similarity Trend Parameter Relationship Meaning
COSINE Higher values more similar radius < distance <= range_filter radius is lower bound, range_filter is upper bound
IP Higher values more similar radius < distance <= range_filter Same as above
L2 Lower values more similar range_filter <= distance < radius range_filter is lower bound, radius is upper bound
JACCARD Lower values more similar range_filter <= distance < radius Same as above
HAMMING Lower values more similar range_filter <= distance < radius Same as above

Memory mnemonic:

  • COSINE/IP: Higher values similar → radius (small) < distance <= range_filter (large)
  • L2/JACCARD/HAMMING: Lower values similar → range_filter (small) <= distance < radius (large)

1.3 Code Example 

def range_search_demo():
    """
    Range search: Limit similarity interval, return only results within interval

    Parameter position: radius and range_filter go in search_params["params"]
    Note: Different metric_type parameter size relationships differ, see table above
    """
    query_vector = [float(random.uniform(-1, 1)) for _ in range(256)]

    # ── COSINE distance: radius < distance <= range_filter ──
    # Target: Books with similarity 0.8 ~ 0.95 (high similarity interval)
    result = client.search(
        collection_name=collection_name,
        data=[query_vector],
        anns_field="content_vector",
        limit=3,
        search_params={
            "metric_type": "COSINE",
            "params": {
                "radius": 0.8,        # Outer circle lower bound (COSINE lower similarity bound)
                "range_filter": 0.95  # Inner circle upper bound (COSINE upper similarity bound)
            }
        },
        output_fields=["book_id", "title", "rating"]
    )

    print(f"High similarity search results (similarity 0.8~0.95): Total {len(result[0])} records")
    for i, hit in enumerate(result[0]):
        print(f"  {i+1}. {hit['entity']['title']}, Similarity: {hit['distance']:.4f}")

    # ── Medium similarity interval ──
    result2 = client.search(
        collection_name=collection_name,
        data=[query_vector],
        anns_field="content_vector",
        limit=3,
        search_params={
            "metric_type": "COSINE",
            "params": {
                "radius": 0.5,
                "range_filter": 0.7   # Similarity 0.5~0.7
            }
        },
        output_fields=["book_id", "title"]
    )
    print(f"Medium similarity search results (similarity 0.5~0.7): Total {len(result2[0])} records")

1.4 Common Scenario Parameter Reference (COSINE)

Scenario radius range_filter Description
Deduplication detection 0.95 1.0 Find almost identical content
High similarity recommendation 0.8 0.95 Strongly related content
Medium similarity recommendation 0.5 0.8 Related but differentiated
Loose matching 0.3 0.6 Broadly related content

⚠️ Note: If parameters set too strictly (interval too narrow), may return 0 results. Recommended to test with loose intervals first, then gradually tighten.

II. Grouping Search

2.1 Core Concepts

Normal ANN search may return many results from the same “category”, causing result homogenization. Grouping search groups by specified field, returns only N most similar from each group, ensuring result diversity.

Typical scenario: RAG document retrieval

Question: A long document is split into 100 chunks stored in Milvus
      Normal search top-5 might all be different segments of same document

Grouping search: Group by doc_id, return 1 from each group
          → top-5 from 5 different documents, more diverse results

2.2 Three Key Parameters

Parameter Type Description
group_by_field str Which field to group by, supports BOOL/INT/VARCHAR/ARRAY, does not support FLOAT
group_size int Maximum records per group, default 1
strict_group_size bool Whether to strictly guarantee group_size records per group, default False

limit meaning changed:

  • Normal search: limit = total entity count returned
  • Grouping search: limit = number of groups returned, total entities ≈ limit × group_size

2.3 Code Example 

def grouping_search_demo():
    """
    Grouping search: Group by field, avoid result homogenization

    book_collection grouped by is_available (BOOL):
    - True group: Most similar among available books
    - False group: Most similar among unavailable books
    """
    query_vector = [float(random.uniform(-1, 1)) for _ in range(256)]

    # ── Basic grouping: Return 1 from each group ──
    result = client.search(
        collection_name=collection_name,
        data=[query_vector],
        anns_field="content_vector",
        limit=2,                          # Return 2 groups (available/unavailable)
        group_by_field="is_available",    # Group by availability
        group_size=1,                     # Return 1 from each group
        output_fields=["book_id", "title", "is_available"]
    )
    # Maximum return 2×1=2 records

    print(f"Basic grouping search results: Total {len(result[0])} records")
    for hit in result[0]:
        entity = hit['entity']
        status = "Available" if entity['is_available'] else "Unavailable"
        print(f"  [{status}] {entity['title']}, Similarity: {hit['distance']:.4f}")

    # ── Multi-record grouping: Return 2 from each group ──
    result2 = client.search(
        collection_name=collection_name,
        data=[query_vector],
        anns_field="content_vector",
        limit=2,                          # 2 groups
        group_by_field="is_available",
        group_size=2,                     # Return 2 from each group
        strict_group_size=True,           # Strictly guarantee 2 per group (may return fewer if data insufficient)
        output_fields=["book_id", "title", "is_available"]
    )
    # Maximum return 2×2=4 records

    print(f"\nMulti-record grouping search results: Total {len(result2[0])} records")
    for hit in result2[0]:
        entity = hit['entity']
        status = "Available" if entity['is_available'] else "Unavailable"
        print(f"  [{status}] {entity['title']}, Similarity: {hit['distance']:.4f}")

2.4 Difference of strict_group_size 

# strict_group_size=False (default)
# Prioritize returning limit groups, each group tries to return group_size records
# With uneven data distribution, some groups may have fewer than group_size records
# Better performance

# strict_group_size=True
# Strictly guarantee group_size records per group (except groups with insufficient data)
# Performance may degrade with uneven data distribution
# Suitable for scenarios requiring strict quantity control per group

2.5 Supported Index Types

Grouping search only supports these indexes: FLAT, IVF_FLAT, IVF_SQ8, HNSW, HNSW_PQ, HNSW_PRQ, HNSW_SQ, DISKANN, SPARSE_INVERTED_INDEX

⚠️ Does not support FLOAT type field grouping, grouping by rating (FLOAT) will cause errors, need to use INT or VARCHAR fields instead.

III. Hybrid Search

3.1 Core Concepts

Hybrid search allows simultaneous search across multiple vector fields, then merges results using reranking strategies.

Why do we need hybrid search?

Limitations of single vector:

  • Dense vectors: Good at semantic understanding, weak at precise keyword matching
  • Sparse vectors: Good at keyword matching, lack semantic understanding

Hybrid search combines both advantages, typical combinations:

Dense vector (semantics) + Sparse vector (keywords) → More accurate text retrieval
Text vector + Image vector → Multimodal search
Multiple text field vectors → Comprehensive multi-dimensional information

3.2 Three-step Process 

Step 1: Create AnnSearchRequest for each vector field
Step 2: Configure reranking strategy (RRF or Weighted)
Step 3: Call hybrid_search() to execute

3.3 Code Example 

def hybrid_vector_search_demo():
    """
    Hybrid search: Simultaneously search content_vector (dense) and keyword_vector (sparse)
    Use RRF reranking strategy to merge results
    """
    from pymilvus import AnnSearchRequest, Function, FunctionType

    query_vector = [float(random.uniform(-1, 1)) for _ in range(256)]

    # ── Step 1: Create independent search requests for each vector field ──

    # Dense vector search request (semantic similarity)
    content_search_req = AnnSearchRequest(
        data=[query_vector],           # Query vector, must be list
        anns_field="content_vector",   # Target vector field
        param={"metric_type": "COSINE", "params": {"nprobe": 10}},
        limit=3                        # Candidate count per sub-request
        # expr="is_available == True"  # Optional: Sub-requests also support filter conditions
    )

    # Sparse vector search request (keyword matching)
    # Sparse vectors represented as dictionaries: {dimension_index: weight_value}
    keyword_query = {0: 0.8, 10: 0.6, 25: 0.4}
    keyword_search_req = AnnSearchRequest(
        data=[keyword_query],
        anns_field="keyword_vector",
        param={"metric_type": "IP", "params": {}},  # Sparse vectors use IP
        limit=3
    )

    # ── Step 2: Configure reranking strategy ──

    # RRF (Reciprocal Rank Fusion): No manual weight adjustment, suitable when unsure of route weights
    ranker = Function(
        name="rrf",
        input_field_names=[],          # Must be empty list, cannot fill vector field names
        function_type=FunctionType.RERANK,
        params={
            "reranker": "rrf",
            "k": 100                   # Control ranking decay speed, default 100, usually no need to change
        }
    )

    # ── Step 3: Execute hybrid search ──
    result = client.hybrid_search(
        collection_name=collection_name,
        reqs=[content_search_req, keyword_search_req],  # All sub-search requests
        ranker=ranker,
        limit=3,                       # Total records returned
        output_fields=["book_id", "title", "rating"]
        # filter="is_available == True"  # Optional: Filter final results
    )

    print(f"Hybrid search results: Total {len(result[0])} records")
    for i, hit in enumerate(result[0]):
        entity = hit['entity']
        print(f"  {i+1}. ID: {entity['book_id']}, Title: {entity['title']}, "
              f"Rating: {entity['rating']}, Combined Score: {hit['distance']:.4f}")

3.4 Two Reranking Strategy Comparison

RRF (Reciprocal Rank Fusion)

ranker = Function(
    name="rrf",
    input_field_names=[],
    function_type=FunctionType.RERANK,
    params={"reranker": "rrf", "k": 100}
)
# Principle: Combined score = Σ 1/(k + rank_i)
# Features: No weight adjustment needed, all search routes equal
# Application: Uncertain importance of routes, or routes equally important

Weighted (Weighted Fusion)

ranker = Function(
    name="weighted",
    input_field_names=[],
    function_type=FunctionType.RERANK,
    params={
        "reranker": "weighted",
        "weights": [0.7, 0.3]   # Weights for each request in reqs list, sum to 1
    }
)
# Features: Can manually control weight of each search route
# Application: Clearly know which route is more important
# Note: weights list length must match reqs list length

3.5 Hybrid Search vs Normal Search

Comparison Item Normal Search search() Hybrid Search hybrid_search()
Vector Fields 1 field Multiple fields
Entry Function client.search() client.hybrid_search()
Sub-requests None AnnSearchRequest list
Reranking None ranker parameter (RRF / Weighted)
Filter Conditions filter parameter Sub-requests or final filter

3.6 Common Pitfalls 

# ❌ Wrong: input_field_names filled with vector field names
ranker = Function(
    input_field_names=["content_vector", "keyword_vector"],  # Wrong!
    ...
)

# ✅ Correct: Must be empty list
ranker = Function(
    input_field_names=[],  # Must be empty list
    ...
)

# ❌ Wrong: Each AnnSearchRequest's data passed multiple vectors
AnnSearchRequest(data=[vec1, vec2], ...)  # Each sub-request only supports 1 query vector

# ✅ Correct
AnnSearchRequest(data=[vec1], ...)

IV. Summary of Three Advanced Search Comparisons

Search Type Problem Solved Key Parameters Typical Scenarios
Range Search Control similarity interval, filter overly close or distant results radius + range_filter Deduplication, recommendation diversity control
Grouping Search Avoid result homogenization, ensure diversity group_by_field + group_size RAG document retrieval, multi-category recommendation
Hybrid Search Fuse multi-route search results AnnSearchRequest + ranker Dense + sparse joint retrieval, multimodal search
