Complete Guide to Installing Milvus Standalone Mode and Common Problem Solutions

Introduction

Milvus is an open-source vector database designed specifically for handling large-scale vector data, supporting vector similarity search, real-time queries, and other features. The Standalone mode is Milvus’s single-machine embedded deployment method, suitable for learning, testing, and small-scale applications. It runs using Docker containers with embedded etcd as metadata storage, requiring no external dependencies.

During the actual installation process, I encountered multiple issues including non-existent image tags, network connection refusals, container permission issues causing startup failures (Exited 134), and scripts getting stuck in waiting states. This article is based on real installation experience on a VMware Ubuntu virtual machine on February 24, 2026, providing complete steps and systematically summarizing problems and solutions. The host machine configuration is Intel Core i9-12900H + 16 GB RAM.

Environment Preparation

System Requirements

  • Ubuntu 20.04 or higher version (22.04 LTS recommended)
  • Docker 20.10 or higher version
  • Minimum VM configuration recommendation: 8-12 GB RAM, 4-8 vCPU (recommended 1 processor √ó 6-8 cores)
  • Network: Domestic mirror accelerator recommended

Configure Docker Image Accelerator

Edit /etc/docker/daemon.json:

{
  "registry-mirrors": [
    "https://docker.xuanyuan.me",
    "https://mirror.ccs.tencentyun.com",
    "https://hub-mirror.c.163.com",
    "https://docker.mirrors.ustc.edu.cn"
  ]
}

Restart Docker:

sudo systemctl daemon-reload
sudo systemctl restart docker

Verify accelerator configuration:

sudo docker info | grep -A 10 "Registry Mirrors"

Installation Steps

1. Download Official Standalone Embedded Startup Script

wget https://raw.githubusercontent.com/milvus-io/milvus/master/scripts/standalone_embed.sh
chmod +x standalone_embed.sh

2. Modify Image Tag (v2.6.11 Not Pushed, Use Latest Available Build from Master Branch)

Edit standalone_embed.sh, find the image line in the docker run command, change:

milvusdb/milvus:v2.6.11 \

to:

milvusdb/milvus:master-20260224-091b147e-amd64 \

or a more stable previous day version:

milvusdb/milvus:master-20260223-49bf0b63-amd64 \

Save the file.

3. Create and Set Data Directory Permissions (Most Critical Step, Avoid Permission Denial)

# Delete old directory (prevent residual permission errors)
rm -rf ./volumes/milvus

# Create new directory
mkdir -p ./volumes/milvus

# Set owner and permissions (ensure non-root user in container can write)
sudo chown -R $USER:$USER ./volumes/milvus
chmod -R 775 ./volumes/milvus

# Verify permissions (should show drwxrwxr-x ai-stu ai-stu)
ls -ld ./volumes/milvus

4. Start Milvus

bash standalone_embed.sh start

5. Verify Success

Wait for script output “Start successfully.”

Check container status:

sudo docker ps

Should show Up ... (healthy)

Test health endpoint:

curl http://localhost:19530/healthz

Should return "ok"

Common Problems Summary and Solutions

Problem 1: Image Pull Failure (not found / connection refused)

Phenomenon:

  • Unable to find image ... not found or dial tcp ... connection refused

Cause:

  • v2.6.11 tag not yet pushed to Docker Hub
  • Domestic network access to registry-1.docker.io is restricted

Solutions:

  1. Use master branch dated build tags (such as master-20260224-091b147e-amd64)
  2. Configure domestic accelerators (Xuanyuan, Alibaba Cloud, NetEase, etc.)
  3. Offline transfer method: pull + save on accessible network machine:
docker pull milvusdb/milvus:master-20260224-091b147e-amd64
docker save -o milvus.tar milvusdb/milvus:master-20260224-091b147e-amd64
gzip milvus.tar

Load after transferring to server:

gunzip milvus.tar.gz
sudo docker load -i milvus.tar

Problem 2: Container Exits Immediately After Startup (Exited 134)

Phenomenon:

  • sudo docker ps -a shows Exited (134)

Cause:

  • SIGABRT signal, internal process abort
  • Usually caused by permission denial, initialization failure, or insufficient memory

Solutions:

  1. Check logs:
    sudo docker logs milvus-standalone --tail 100
  2. Increase VM memory to 8-12 GB
  3. Clean and reset permissions (see step 3)
  4. Try more stable tags (such as master-20260223-49bf0b63-amd64)

Problem 3: Permission Denied (mkdir /var/lib/milvus/data/: permission denied)

Phenomenon:

  • Logs show [FATAL] ... permission denied and panic

Cause:

  • Host ./volumes/milvus directory lacks sufficient permissions
  • Non-root user in container cannot create subdirectories

Solutions:

  1. Must delete old directory first:
    rm -rf ./volumes/milvus
  2. Recreate and set permissions (step 3)
  3. Recommend using 775 permissions to ensure group writability

Problem 4: Script Stuck at “Wait for Milvus Starting…”

Phenomenon:

  • Script shows no output for extended period

Cause:

  • Container hasn’t reached healthy state (crashed or health check failed)

Solutions:

  1. Check status in new terminal:
    sudo docker ps -a
  2. View real-time logs:
    sudo docker logs -f milvus-standalone
  3. Manually test health endpoint:
    curl http://localhost:19530/healthz
  4. Clean data volumes and retry
