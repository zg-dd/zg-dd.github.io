I. Comprehensive Comparison of Milvus Five Query Methods

Milvus provides 5 query methods, divided into two major categories: scalar queries (3 methods) and vector searches (2 methods).

1.2 Scalar Query vs Vector Search

Scalar Queries (Get / Query / QueryIterator):

  • No vector calculations involved, fast query speed
  • Based on primary key index or scalar field index
  • Suitable for precise lookups, conditional filtering, data export
  • Results have no similarity sorting

Vector Searches (Search / SearchIterator):

  • Involves vector similarity calculation (ANN search)
  • Based on vector indexes (HNSW, IVF, etc.)
  • Suitable for semantic search, recommendation systems, similar content lookup
  • Results sorted by similarity, includes distance field

1.3 Method Selection Decision Tree 

Need to query data?
├── Based on vector similarity?
│   ├── Yes → Vector Search
│   │   ├── Result count < 16384?
│   │   │   └── Yes → Use Search
│   │   └── Result count > 16384?
│   │       └── Yes → Use SearchIterator
│   └── No → Scalar Query
│       ├── Known primary key?
│       │   └── Yes → Use Get (Fastest)
│       └── No → Conditional filtering, judge data volume
│           ├── Result count < 10,000 → Use Query
│           └── Result count > 10,000 → Use QueryIterator

Scalar queries refer to query methods that do not involve vector similarity calculations, only performing data retrieval through primary keys or scalar field conditions. This section focuses on the three scalar query methods: Get, Query, and QueryIterator.

II. Get Method: Query by Primary Key

2.1 Core Features

  • Fastest query method: Direct lookup through primary key index, time complexity O(1)
  • No vector calculations involved: Pure scalar field query
  • Precise matching: Must provide accurate primary key values

2.2 Code Example 

from pymilvus import MilvusClient

client = MilvusClient(uri="http://localhost:19530")

# Get entities by primary key
result = client.get(
    collection_name="book_collection",
    ids=[1, 5, 10],  # Primary key list
    output_fields=["book_id", "title", "isbn", "rating"]
)

# Iterate through results
for record in result:
    print(f"ID: {record['book_id']}, Title: {record['title']}")

2.3 Parameter Explanation

Parameter Type Required Description
collection_name str Yes Collection name
ids List Yes Primary key list
output_fields List[str] No List of fields to return, defaults to primary key only
partition_names List[str] No Limit query to specific partitions

2.4 Return Result Structure

III. Query Method: Conditional Filtering Query

3.1 Core Features

  • Flexible conditional filtering: Supports rich filter expressions
  • Controllable return quantity: Limit result count via limit parameter
  • No vector calculations involved: Pure scalar field query

3.2 Basic Usage 

# Simple condition query
result = client.query(
    collection_name="book_collection",
    filter="is_available == True",
    output_fields=["book_id", "title", "is_available"],
    limit=10
)

# Composite condition query
result = client.query(
    collection_name="book_collection",
    filter="is_available == True and rating >= 4.0",
    output_fields=["book_id", "title", "rating"],
    limit=5
)

3.3 Parameter Explanation

Parameter Type Required Description
collection_name str Yes Collection name
filter str Yes Filter expression (empty string means query all)
output_fields List[str] No List of fields to return
limit int No Maximum number of results, default 16384
partition_names List[str] No Limit query to specific partitions
filter_params dict No Filter template parameters (see Section 3.6)

IV. Detailed Filtering Expressions

4.1 Comparison Operators 

# Equal
filter = "is_available == True"
filter = "rating == 4.5"

# Not equal
filter = "rating != 3.0"

# Greater than/Less than
filter = "rating > 4.0"
filter = "price < 100.0"
filter = "page_count >= 200"
filter = "rating <= 5.0"

4.2 Range Operators 

# IN operator
filter = "rating in [4.0, 4.5, 5.0]"
filter = "book_id in [1, 2, 3, 5, 8]"

# LIKE operator (fuzzy matching)
filter = 'title like "%Python%"'    # Contains "Python"
filter = 'title like "Python%"'     # Starts with "Python"
filter = 'title like "%Programming"' # Ends with "Programming"

4.3 Logical Operators 

# AND
filter = "is_available == True and rating >= 4.0"

# OR
filter = 'title like "%Python%" or title like "%AI%"'

# NOT
filter = "not is_available == True"

# Complex combination
filter = """
(is_available == True and rating >= 4.0) and
(price < 150.0 or page_count > 300)
"""

4.4 Null Value Checking 

# Not null
filter = "details is not null"

# Null value
filter = "description is null"

4.5 Arithmetic Operators 

# Modulo (find even ID)
filter = "book_id % 2 == 0"

# Multiplication
filter = "price * 0.8 < 100"

# Division
filter = "page_count / 100 > 2"

# Addition/Subtraction
filter = "rating + 0.5 >= 5.0"

4.6 Filtering Templates

Why do we need filtering templates?

  • Improve performance when processing CJK (Chinese/Japanese/Korean) characters
  • Enhance code readability and security
  • Query engine can cache parsed expression structures

Basic Usage:

# Define template (using {param_name} placeholder)
template_filter = "rating >= {min_rating} and is_available == {available}"

# Pass parameters
template_params = {
    "min_rating": 4.0,
    "available": True
}

# Execute query
result = client.query(
    collection_name="book_collection",
    filter=template_filter,
    filter_params=template_params,
    output_fields=["book_id", "title", "rating"],
    limit=10
)

Array IN Operation Template:

# Template definition
in_filter = "rating in {ratings}"

# Parameter passing
in_params = {"ratings": [4.0, 4.5, 5.0]}

result = client.query(
    collection_name="book_collection",
    filter=in_filter,
    filter_params=in_params,
    output_fields=["book_id", "title", "rating"],
    limit=10
)

Notes:

  • Placeholder names must exactly match keys in filter_params
  • Ensure parameter values match field data types
  • Requires Milvus 2.4+ version support

V. QueryIterator Method: Pagination Query

5.1 Core Features

  • Memory-friendly: Stream processing, doesn’t load all results at once
  • Suitable for large data volumes: Batch retrieval avoids memory overflow
  • No vector calculations involved: Pure scalar field query

5.2 Code Example 

# Create query iterator
iterator = client.query_iterator(
    collection_name="book_collection",
    filter="is_available == True",
    output_fields=["book_id", "title", "is_available"],
    batch_size=50  # Return 50 records per batch
)

# Retrieve results in batches
all_results = []
while True:
    batch_result = iterator.next()
    if not batch_result:
        break

    all_results.extend(batch_result)
    print(f"This batch returned {len(batch_result)} records")

print(f"Total retrieved {len(all_results)} records")

5.3 Parameter Explanation

Parameter Type Required Description
collection_name str Yes Collection name
filter str Yes Filter expression
output_fields List[str] No List of fields to return
batch_size int Yes Number of records per batch
partition_names List[str] No Limit query to specific partitions

5.4 Usage Scenarios

  • Need to export large amounts of data
  • Need to traverse all matching records in collection
  • Memory-constrained environments
  • Need real-time processing of query results (streaming processing)

VI. Special Query Techniques

6.1 Count Entity Numbers 

# Count total in collection
result = client.query(
    collection_name="book_collection",
    filter="",
    output_fields=["count(*)"]
)
print(result)  # [{'count(*)': 2000}]

# Count satisfying conditions
result = client.query(
    collection_name="book_collection",
    filter="rating >= 4.5",
    output_fields=["count(*)"]
)

6.2 Query All Data 

# Empty filter expression queries all
result = client.query(
    collection_name="book_collection",
    filter="",
    output_fields=["book_id", "title"],
    limit=100
)

6.3 Partition Query 

# Limit query to specific partition
result = client.query(
    collection_name="book_collection",
    filter="rating >= 4.0",
    partition_names=["partition_2024"],
    output_fields=["book_id", "title"]
)

VII. Method Selection Guide

7.1 Decision Tree 

Need to query data?
├── Known primary key?
│   └── Yes → Use Get (Fastest)
└── No → Need conditional filtering
    ├── Small data volume (< 10,000 records)?
    │   └── Yes → Use Query
    └── Large data volume (> 10,000 records)?
        └── Yes → Use QueryIterator

7.2 Performance Comparison

Method Query Speed Memory Usage Suitable Data Volume
Get Fastest Low Any
Query Fast Medium < 10,000 records
QueryIterator Medium Low (Streaming) Any

VIII. Common Issues

8.1 Query Returns Empty Results

Possible Causes:

  • Filter conditions too strict
  • Collection not loaded into memory
  • Field name spelling errors

Solutions:

# 1. Check collection load status
load_state = client.get_load_state(collection_name="book_collection")
print(load_state)

# 2. Simplify filter conditions for testing
result = client.query(
    collection_name="book_collection",
    filter="",  # Empty filter queries all
    limit=1
)

8.2 QueryIterator High Memory Usage

Cause: batch_size setting too large

Solution:

# Reduce batch_size
iterator = client.query_iterator(
    collection_name="book_collection",
    filter="is_available == True",
    batch_size=10  # Reduce from 100 to 10
)

8.3 Filter Expression Syntax Errors

Common Errors:

# ❌ Wrong: String without quotes
filter = "title == Python"

# ✅ Correct
filter = 'title == "Python"'

# ❌ Wrong: LIKE wildcard position error
filter = 'title like "Python*"'

# ✅ Correct
filter = 'title like "Python%"'
