1.2 Why Do We Need Full-text Search? Isn’t Vector Search Enough?

Plain Explanation: Dense Vectors vs Sparse Vectors

Your understanding is correct: Full-text search (BM25) indeed uses sparse vectors. Let’s illustrate the differences with examples:

Dense Vector Search

Usage: Vectors generated by models like BERT, OpenAI embeddings
Features: Every dimension has values, vectors are "dense"

Example: Search "apple phone"
✅ Can match: "iPhone 13", "iOS device" (understands semantics)
❌ But may match: "Python 3.11 tutorial" (if searching "Python 3.11")
   Because it understands "programming language" semantics, but insensitive to version numbers

Advantages: Understands semantics, “apple phone” = “iPhone” Disadvantages: Insensitive to precise keywords (like version numbers, models)

Sparse Vector Search (BM25)

Usage: BM25 algorithm based on word frequency statistics
Features: Only dimensions with appearing words have values, vectors are "sparse"

Example: Search "Python 3.11 new features"
✅ Precise matching of documents containing "3.11"
✅ Precise matching of keywords "Python", "3.11", "new features"
❌ But doesn't understand "iPhone" and "apple phone" mean the same thing

Advantages: Precise keyword matching, sensitive to version numbers, models, proper nouns Disadvantages: Doesn’t understand semantics, “apple phone” ≠ “iPhone”

Practical Application Scenario Comparison

ScenarioRecommended MethodReason
Search “machine learning beginner tutorial”Dense vectorNeed to understand “beginner”, “tutorial” semantics
Search “Python 3.11.5 bug”Sparse vector (BM25)Version number must match precisely
Search “iPhone 14 Pro Max”Sparse vector (BM25)Model must match precisely
Search “how to improve code quality”Dense vectorNeed to understand intent, match related content
Search “ISBN: 978-7-115-12345-6”Sparse vector (BM25)Precise matching of numbers

Why Does Milvus Support Both?

Best Practice: Hybrid Search

# Use both dense and sparse vectors simultaneously
# Dense vector: Understand semantics
# Sparse vector: Precise keyword matching
# Result fusion: Both semantic understanding and keyword precision

Typical RAG Application Scenario:

  • User asks: “What are the new features in Python 3.11?”
  • Dense vector: Find semantically related “Python new version introduction” documents
  • Sparse vector: Precisely match paragraphs containing “3.11”
  • Fused results: Both relevant and precise

Summary:

  • Full-text search (BM25) doesn’t replace vector search, but complements it
  • Dense vectors excel at semantic understanding, sparse vectors excel at precise matching
  • Often used together in practical applications to leverage respective advantages

II. LIKE Operator: Character Pattern Matching

2.1 Core Features

  • Simplest text matching method: Character-level pattern matching
  • No tokenization processing: Direct string matching
  • Best performance: No complex text analysis involved
  • No special configuration required: Any VARCHAR field supports

2.2 Wildcard Explanation

WildcardMeaningExampleMatching Results
%Matches any length characters (including 0)"Python%"Python, Python programming, Python3
%Matches any length characters"%programming"Python programming, Java programming
%Matches any length characters"%machine learning%"Deep learning and machine learning, machine learning practice

2.3 Code Example

from pymilvus import MilvusClient

client = MilvusClient(uri="http://localhost:19530")

# 1. Prefix matching
result = client.query(
    collection_name="book_collection",
    filter='title LIKE "Python%"',
    output_fields=["book_id", "title"],
    limit=5
)

# 2. Suffix matching
result = client.query(
    collection_name="book_collection",
    filter='title LIKE "%programming"',
    output_fields=["book_id", "title"],
    limit=5
)

# 3. Infix matching (contains)
result = client.query(
    collection_name="book_collection",
    filter='title LIKE "%machine learning%"',
    output_fields=["book_id", "title"],
    limit=5
)

2.4 Application Scenarios

  • Simple prefix, suffix, infix matching
  • Precise string lookup without tokenization
  • High performance requirement scenarios
  • English words or fixed format matching

2.5 Limitations

  • No smart tokenization support
  • Cannot handle synonyms
  • Unfriendly to Chinese tokenization
  • No relevance scoring

III. TEXT_MATCH: Keyword Matching

3.1 Core Features

  • Smart tokenization: Automatically tokenize query text
  • OR logic: Default uses OR logic to match multiple keywords
  • Supports logical combinations: Can combine with AND, OR, NOT
  • Field configuration required: Field must enable enable_match=True

3.2 Basic Usage

# Single keyword matching
result = client.query(
    collection_name="book_collection",
    filter="TEXT_MATCH(title, 'Python')",
    output_fields=["book_id", "title"],
    limit=5
)

# Multi-keyword matching (OR logic)
result = client.query(
    collection_name="book_collection",
    filter="TEXT_MATCH(title, 'Python machine learning')",
    output_fields=["book_id", "title"],
    limit=5
)
# Matches records containing "Python" OR "machine learning"

3.3 Logical Combinations

# AND logic: Contain multiple keywords simultaneously
result = client.query(
    collection_name="book_collection",
    filter="TEXT_MATCH(title, 'Python') AND TEXT_MATCH(title, 'programming')",
    output_fields=["book_id", "title"],
    limit=5
)

# NOT logic: Exclude specific keywords
result = client.query(
    collection_name="book_collection",
    filter="TEXT_MATCH(title, 'Python') AND NOT TEXT_MATCH(title, 'Web')",
    output_fields=["book_id", "title"],
    limit=5
)

# Combine with scalar filtering
result = client.query(
    collection_name="book_collection",
    filter="TEXT_MATCH(title, 'Python') AND rating >= 4.5",
    output_fields=["book_id", "title", "rating"],
    limit=5
)

3.4 Application Scenarios

  • Keyword retrieval
  • Complex queries requiring logical combinations
  • Chinese tokenization scenarios
  • Queries combining with scalar filtering

IV. PHRASE_MATCH: Phrase Matching

4.1 Core Features

  • Order-sensitive: Strictly match word order
  • Precise phrase matching: Suitable for finding specific phrases
  • Supports slop parameter: Allows flexibility between words
  • Field configuration required: Field must enable enable_match=True

4.2 slop Parameter Details

slop ValueMeaningSample QueryMatching Results
0 (default)Exact match, word order and position completely consistent“machine learning”✓ “machine learning”✗ “machine boosts learning”
1Allow one word gap or position adjustment“machine learning”✓ “machine learning”✓ “machine boosts learning”
2Allow two word gaps or word order reversal“machine learning”✓ “learning machine”✓ “machine quickly boosts learning”

4.3 Code Example

# Exact phrase matching (slop=0, default)
result = client.query(
    collection_name="book_collection",
    filter="PHRASE_MATCH(title, 'Python programming')",
    output_fields=["book_id", "title"],
    limit=5
)

# Fuzzy matching with slop parameter
result = client.query(
    collection_name="book_collection",
    filter="PHRASE_MATCH(title, 'machine learning', 2)",
    output_fields=["book_id", "title"],
    limit=5
)

# Combine with scalar filtering
result = client.query(
    collection_name="book_collection",
    filter="PHRASE_MATCH(title, 'Python') AND is_available == True",
    output_fields=["book_id", "title", "is_available"],
    limit=5
)

4.4 Application Scenarios

  • Scenarios requiring precise phrase matching
  • Queries where word order matters (proper nouns, fixed collocations)
  • Fuzzy matching controlling word spacing
  • Finding quotations or specific expressions

V. Full-text Search (BM25)

5.1 Core Features

  • BM25 Algorithm: Probability retrieval model-based relevance scoring
  • Automatic vectorization: Text automatically converted to sparse vectors, no manual generation needed
  • Relevance sorting: Results returned sorted by BM25 scores
  • Suitable for RAG scenarios: Particularly suitable for Retrieval-Augmented Generation (RAG) applications

5.2 BM25 Algorithm Introduction

BM25 (Best Matching 25) is a ranking function for information retrieval, considering these factors:

  • Term Frequency (TF): Frequency of keywords appearing in documents
  • Inverse Document Frequency (IDF): Rarity of keywords across the entire collection
  • Document Length Normalization: Avoid bias toward longer documents

5.3 What Do You Need to Do? What Does Milvus Do?

Important Note: Sparse vectors are automatically generated!

Many people mistakenly think they need to manually generate both dense and sparse vectors, but actually:

Data Preparation Division of Labor

Data TypeDo You Need to GenerateExplanation
Raw Text✅ YesStore file content, for returning results and generating sparse vectors
Dense Vector✅ YesGenerated using OpenAI/BERT models, for semantic search
Sparse Vector (BM25)❌ NoAutomatically generated by Milvus (configure BM25 function)

Actual Workflow Example

# ========== Step 1: Read file content ==========
file_content = "Python 3.11 introduces new exception group functionality, enhancing error handling capabilities..."

# ========== Step 2: Generate dense vector (you need to do) ==========
import openai

dense_vector = openai.Embedding.create(
    input=file_content,
    model="text-embedding-ada-002"
)['data'][0]['embedding']

# ========== Step 3: Insert into Milvus ==========
# Only need to provide: raw text + dense vector
# Sparse vector will be automatically generated!
client.insert('my_collection', [{
    "text": file_content,           # Raw text
    "dense_vector": dense_vector,   # Your generated dense vector
    # sparse_vector not needed, automatically generated by Milvus!
}])

# ========== Step 4: Hybrid search ==========
query = "Python 3.11 new features"

# Generate query dense vector (you need to do)
query_dense = openai.Embedding.create(
    input=query,
    model="text-embedding-ada-002"
)['data'][0]['embedding']

# Dense vector search request
dense_req = AnnSearchRequest(
    data=[query_dense],
    anns_field="dense_vector",
    param={"metric_type": "COSINE"},
    limit=5
)

# Sparse vector search request
# Note: Pass text directly, Milvus automatically converts to sparse vector!
sparse_req = AnnSearchRequest(
    data=[query],  # Pass query text directly
    anns_field="sparse_vector",
    param={"metric_type": "BM25"},
    limit=5
)

# Execute hybrid search
result = client.hybrid_search(
    collection_name='my_collection',
    reqs=[dense_req, sparse_req],
    ranker=rrf_ranker,
    limit=10
)

Key Points Summary

✅ You only need to care about dense vector generation (using OpenAI, BERT, sentence-transformers, etc.)

✅ Sparse vectors completely handled by Milvus internally (automatically generated after configuring BM25 function)

✅ Don’t need to manually generate sparse vectors during search (pass text directly, Milvus automatically converts)

❌ Don’t need to implement BM25 algorithm yourself

❌ Don’t need to manually calculate term frequency and IDF

This is why Milvus’s full-text search is powerful: It encapsulates complex BM25 calculations, you only need to provide raw text!

5.4 Complete Configuration Process

Step 1: Create Schema Supporting Full-text Search

Need to define three required fields: primary key, text field, sparse vector field

from pymilvus import MilvusClient, DataType, Function, FunctionType

client = MilvusClient(
    uri="http://localhost:19530",
    token="root:Milvus"
)

# Create Schema
schema = client.create_schema()

# 1. Primary key field
schema.add_field(
    field_name="id",
    datatype=DataType.INT64,
    is_primary=True,
    auto_id=True
)

# 2. Text field (enable tokenizer)
schema.add_field(
    field_name="text",
    datatype=DataType.VARCHAR,
    max_length=1000,
    enable_analyzer=True  # Must enable tokenizer
)

# 3. Sparse vector field (store BM25 vectors)
schema.add_field(
    field_name="sparse",
    datatype=DataType.SPARSE_FLOAT_VECTOR
)

Step 2: Define BM25 Function

BM25 function automatically converts text to sparse vectors:

# Define BM25 function
bm25_function = Function(
    name="text_bm25_emb",              # Function name
    input_field_names=["text"],         # Input: text field
    output_field_names=["sparse"],      # Output: sparse vector field
    function_type=FunctionType.BM25,    # Function type: BM25
)

# Add function to Schema
schema.add_function(bm25_function)

Step 3: Configure Sparse Vector Index

# Prepare index parameters
index_params = client.prepare_index_params()

# Add index for sparse vector field
index_params.add_index(
    field_name="sparse",
    index_type="SPARSE_INVERTED_INDEX",  # Sparse inverted index
    metric_type="BM25",                   # Metric type: BM25
    params={
        "inverted_index_algo": "DAAT_MAXSCORE",  # Inverted index algorithm
        "bm25_k1": 1.2,                          # BM25 parameter k1 (term frequency saturation)
        "bm25_b": 0.75                           # BM25 parameter b (length normalization)
    }
)

BM25 Parameter Explanation:

  • bm25_k1 (default 1.2): Controls term frequency saturation, higher values mean greater TF impact
  • bm25_b (default 0.75): Controls document length normalization, 0 means no normalization, 1 means full normalization

Step 4: Create Collection

# Create collection
client.create_collection(
    collection_name='my_collection',
    schema=schema,
    index_params=index_params
)

Step 5: Insert Text Data

Insert raw text directly, BM25 function automatically generates sparse vectors:

# Insert text data (no need to manually generate vectors)
client.insert('my_collection', [
    {'text': 'information retrieval is a field of study.'},
    {'text': 'information retrieval focuses on finding relevant information in large datasets.'},
    {'text': 'data mining and information retrieval overlap in research.'},
])

Step 6: Execute Full-text Search

# Execute full-text search
search_params = {
    "metric_type": "BM25",
    "params": {}
}

results = client.search(
    collection_name="my_collection",
    data=["What is information retrieval?"],  # Query text
    anns_field="sparse",                      # Search sparse vector field
    search_params=search_params,
    limit=5,
    output_fields=["text"]
)

# Output results
for i, hit in enumerate(results[0]):
    print(f"{i+1}. Text: {hit['entity']['text']}")
    print(f"   Relevance Score: {hit['distance']:.4f}\n")

5.5 Complete Example

from pymilvus import MilvusClient, DataType, Function, FunctionType

# 1. Connect to Milvus
client = MilvusClient(uri="http://localhost:19530")

# 2. Create Schema
schema = client.create_schema()
schema.add_field(field_name="id", datatype=DataType.INT64, is_primary=True, auto_id=True)
schema.add_field(field_name="text", datatype=DataType.VARCHAR, max_length=1000, enable_analyzer=True)
schema.add_field(field_name="sparse", datatype=DataType.SPARSE_FLOAT_VECTOR)

# 3. Add BM25 function
bm25_function = Function(
    name="text_bm25_emb",
    input_field_names=["text"],
    output_field_names=["sparse"],
    function_type=FunctionType.BM25,
)
schema.add_function(bm25_function)

# 4. Configure index
index_params = client.prepare_index_params()
index_params.add_index(
    field_name="sparse",
    index_type="SPARSE_INVERTED_INDEX",
    metric_type="BM25",
    params={"bm25_k1": 1.2, "bm25_b": 0.75}
)

# 5. Create collection
client.create_collection(
    collection_name='full_text_demo',
    schema=schema,
    index_params=index_params
)

# 6. Insert data
client.insert('full_text_demo', [
    {'text': 'Artificial intelligence is transforming the world.'},
    {'text': 'Machine learning is a subset of artificial intelligence.'},
    {'text': 'Deep learning uses neural networks for complex tasks.'},
])

# 7. Execute search
results = client.search(
    collection_name="full_text_demo",
    data=["What is artificial intelligence?"],
    anns_field="sparse",
    search_params={"metric_type": "BM25"},
    limit=3,
    output_fields=["text"]
)

# 8. Output results
for i, hit in enumerate(results[0]):
    print(f"{i+1}. {hit['entity']['text']} (Score: {hit['distance']:.4f})")

VI. Method Selection Guide

6.1 Decision Tree

Need text retrieval?
├── Simple character matching?
│   └── Yes → Use LIKE (best performance)
└── No → Need tokenization
    ├── Precise phrase matching?
    │   └── Yes → Use PHRASE_MATCH
    ├── Keyword retrieval?
    │   └── Yes → Use TEXT_MATCH
    └── Need relevance sorting?
        └── Yes → Use full-text search (BM25)

Search articles most relevant to “AI technology”Full-text searchNeed relevance scoring

# 1. Prioritize LIKE (best performance)
# Suitable: Simple prefix/suffix matching
filter='title LIKE "Python%"'

# 2. TEXT_MATCH suitable for most scenarios

# 3. PHRASE_MATCH for precise matching
# Suitable: Scenarios requiring word order preservation
filter="PHRASE_MATCH(title, 'Python programming')"

# 4. Full-text search for complex scenarios

7.2 Field Configuration

To use TEXT_MATCH or PHRASE_MATCH, need to configure fields:

from pymilvus import MilvusClient, DataType

schema = client.create_schema()

# Add field supporting text matching
schema.add_field(
    field_name="title",
    datatype=DataType.VARCHAR,
    max_length=500,
    enable_match=True,        # Enable text matching
    enable_analyzer=True      # Enable tokenizer (optional)
)

7.3 Common Combination Patterns

# Pattern 1: Text matching + scalar filtering
filter="TEXT_MATCH(title, 'Python') AND rating >= 4.5 AND is_available == True"

# Pattern 2: Multi-field text matching
filter="TEXT_MATCH(title, 'Python') OR TEXT_MATCH(description, 'Python')"

# Pattern 3: Phrase matching + exclusion
filter="PHRASE_MATCH(title, 'machine learning') AND NOT TEXT_MATCH(title, 'beginner')"

VIII. Common Issues

8.1 TEXT_MATCH Error

Error Message: field 'title' doesn't support text match

Cause: Field didn’t enable enable_match=True

Solution:

# Configure field when creating collection
schema.add_field(
    field_name="title",
    datatype=DataType.VARCHAR,
    max_length=500,
    enable_analyzer=True, # Enable tokenizer
    enable_match=True  # Must enable
)

8.2 LIKE Doesn’t Match Chinese

Cause: LIKE is character-level matching, Chinese tokenization may cause matching failure

Solutions:

# Solution 1: Use TEXT_MATCH (recommended)
filter="TEXT_MATCH(title, 'machine learning')"

# Solution 2: Use complete string
filter='title LIKE "%machine learning%"'

# Solution 3: Configure tokenizer method
analyzer_params = {
    "type": "english"
}
schema.add_field(
    field_name="title",
    datatype=DataType.VARCHAR,
    max_length=500,
    enable_analyzer=True, # Enable tokenizer
    analyzer_params = analyzer_params,
    enable_match=True  # Must enable
)

8.3 PHRASE_MATCH Doesn’t Match Results

Cause: Strict word order, may fail due to tokenization or word order issues

Solutions:

# Increase slop parameter, allow word gaps
filter="PHRASE_MATCH(title, 'machine learning', 2)"

# Or switch to TEXT_MATCH
filter="TEXT_MATCH(title, 'machine learning')"
