1. Understanding Schema Concept

1.1 What is Schema?

Schema is the metadata definition of a Collection, similar to table structure definition in relational databases.

Milvus ConceptRelational Database ConceptDescription
SchemaTable SchemaMetadata defining data structure
FieldColumnField/column definition
DataTypeData TypeData types (VARCHAR, INT, FLOAT, etc.)
Primary KeyPrimary KeyPrimary key field
Vector Field-Milvus-specific vector field
Dynamic Field-Dynamic field, similar to MongoDB’s flexible Schema

JSON Indexing

JSON indexing suitable scenarios:

  1. Structured Schema with consistent, known keys
  2. Equality and range queries on specific JSON paths
  3. Need for precise control over indexed keys
  4. Efficient storage acceleration for target queries
# Create JSON index
index_params = MilvusClient.prepare_index_params()

index_params.add_index(
    field_name="product_detail",        # JSON field name
    index_type="AUTOINDEX",             # Must be AUTOINDEX or INVERTED
    index_name="idx_product_brand",     # Index name
    params={
        "json_path": 'product_detail["brand"]',  # Top-level key
        "json_cast_type": "VARCHAR"              # Data type
    }
)

# Nested key indexing
index_params.add_index(
    field_name="product_detail",
    index_type="AUTOINDEX",
    index_name="idx_product_cpu",
    params={
        "json_path": 'product_detail["specs"]["cpu"]',  # Nested key
        "json_cast_type": "VARCHAR"
    }
)

# Compare with MySQL 8.0+
# CREATE INDEX idx_product_brand ON products((JSON_EXTRACT(product_detail, '$.brand')));

JSON Shredding Optimization

JSON shredding accelerates queries by converting row-based storage to columnar storage.

Parameter NameDescriptionDefaultAdjustment Suggestions
common.enabledJSONShreddingWhether to enable JSON shreddingfalseSet to true to activate feature
common.usingJSONShreddingForQueryWhether to use shredded data for accelerationtrueSet to false to restore when queries fail
queryNode.mmap.jsonShreddingWhether to use mmap when loadingtrueAdjust for special memory requirements
dataCoord.jsonShreddingMaxColumnsMaximum shredded columns1024Increase when frequent keys exceed limit
dataCoord.jsonShreddingRatioThresholdMinimum occurrence rate threshold0.3Adjust shredding criteria
# milvus.yaml configuration example
common:
  enabledJSONShredding: true
  usingJSONShreddingForQuery: true

queryNode:
  mmap:
    jsonShredding: true

dataCoord:
  jsonShreddingMaxColumns: 1024
  jsonShreddingRatioThreshold: 0.3

schema.add_field(
    field_name="created_at",
    datatype=DataType.TIMESTAMPTZ,
    nullable=False,
    default_value="2025-01-01T00:00:00+00:00"

# Insert data example
data = {
    "product_id": 1,
    "created_at": "2025-05-01T23:59:59+08:00"  # Beijing time
}

# Compare with MySQL
# created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP

Timezone Management Levels

LevelParameterScopePriority
DatabasetimezoneDefault for all collections in databaseLowest
CollectiontimezoneOverride database default timezoneMedium
Query/SearchtimezoneTemporary override for specific operationHighest
# Database level timezone setting
client.alter_database_properties(
    "my_db",
    {"timezone": "Asia/Shanghai"}
)

# Collection level timezone setting (when creating collection)
client.create_collection(
    collection_name="products",
    schema=schema,
    properties={"timezone": "America/New_York"}
)

# Specify timezone temporarily during query
client.query(
    collection_name="products",
    filter="created_at > '2025-01-01T00:00:00+08:00'",
    timezone="Asia/Shanghai"
)

4.6 Struct Arrays (ARRAY of STRUCT)

Struct arrays are used to store complex nested data.

# Product reviews array example
schema.add_field(
    field_name="reviews",
    datatype=DataType.ARRAY,
    element_type=DataType.STRUCT,
    max_capacity=100,
    element_schema={
        "fields": [
            {"name": "user_id", "type": DataType.INT64},
            {"name": "rating", "type": DataType.INT32},
            {"name": "comment", "type": DataType.VARCHAR, "max_length": 500},
            {"name": "review_vector", "type": DataType.FLOAT_VECTOR, "dim": 64},
            {"name": "timestamp", "type": DataType.TIMESTAMPTZ}
        ]
    }
)

# Insert data example
data = {
    "product_id": 1,
    "reviews": [
        {
            "user_id": 1001,
            "rating": 5,
            "comment": "Very good!",
            "review_vector": [0.1, 0.2, ...],  # 64-dimensional vector
            "timestamp": "2025-05-01T10:00:00+08:00"
        },
        {
            "user_id": 1002,
            "rating": 4,
            "comment": "Pretty good",
            "review_vector": [0.3, 0.4, ...],
            "timestamp": "2025-05-02T15:30:00+08:00"
        }
    ]
}

# Compare with MySQL
# Need to create association table reviews
# CREATE TABLE reviews (
#     review_id BIGINT PRIMARY KEY,
#     product_id BIGINT,
#     user_id BIGINT,
#     rating INT,
#     comment VARCHAR(500),
#     timestamp TIMESTAMP,
#     FOREIGN KEY (product_id) REFERENCES products(product_id)

6. Dynamic Fields

6.1 Dynamic Field Concept

Dynamic fields allow inserting fields not defined in Schema, which are automatically stored in the hidden $meta JSON field.

# Enable dynamic fields
schema = client.create_schema(
    enable_dynamic_field=True  # Enable dynamic fields
)

# Define basic fields
schema.add_field(
    field_name="product_id",
    datatype=DataType.INT64,
    is_primary=True,
    auto_id=True
)

schema.add_field(
    field_name="product_vector",
    datatype=DataType.FLOAT_VECTOR,
    dim=128
)

# Can include undefined fields when inserting data
data = {
    "product_id": 1,
    "product_vector": [0.1, 0.2, ...],
    "name": "iPhone 15",           # Not defined in Schema
    "category": "Electronics",     # Not defined in Schema
    "price": 999.99               # Not defined in Schema
}

# Milvus internally stores as:
# {
#   "product_id": 1,
#   "product_vector": [0.1, 0.2, ...],
#   "$meta": {
#     "name": "iPhone 15",
#     "category": "Electronics",
#     "price": 999.99
#   }
# }

6.3 Querying Dynamic Fields

# Direct field name query
results = client.query(
    collection_name="products",
    filter='name == "iPhone 15"',  # Directly use dynamic field name
    output_fields=["product_id", "name", "category", "price"]
)

# Or query through $meta
results = client.query(
    collection_name="products",
    filter='$meta["category"] == "Electronics"',
    output_fields=["product_id", "$meta"]
)

# Compare with MongoDB
# db.products.find({"name": "iPhone 15"})

# HNSW index (recommended for high-precision search)
index_params.add_index(
    field_name="product_embedding",
    index_type="HNSW",

10. Key Points Summary

  1. Schema Flexibility: Milvus supports dynamic fields, more flexible than traditional relational databases
  2. Native Vector Support: Vector fields are first-class citizens, no extra conversion needed
  3. Rich Types: Supports JSON, arrays, structs and other complex types
  4. Timezone Awareness: TIMESTAMPTZ provides native timezone support
  5. Index Optimization: JSON shredding, vector indexing and other optimization techniques
  6. NoSQL Integration: Dynamic fields similar to MongoDB, but retain Schema constraint advantages
