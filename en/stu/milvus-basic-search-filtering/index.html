简体中文

Coverage: Basic ANN search (single vector, batch), filtered search (standard filtering & iterative filtering), pagination search Official Documentation: Basic ANN Search | Filtered Search

I. Core Concept: ANN Search

Milvus uses ANN (Approximate Nearest Neighbor) approximate nearest neighbor search, not exact kNN search.

Comparison ItemkNNANN
PrincipleTraverse all vectorsLocate candidate subset based on index files
Accuracy100% exactApproximate, adjustable
PerformanceSlow, high resource consumptionFast, suitable for large-scale data
Application ScenariosSmall datasetsProduction environment

Search Process:

  1. Quickly locate candidate subset based on index files
  2. Calculate similarity using specified distance metric (metric_type)
  3. Sort by similarity, return top-K results

Distance Metrics

metric_typeSimilarity TrendValue RangeApplication Scenarios
COSINEHigher values more similar[-1, 1]Text semantic search (most common)
IP (Inner Product)Higher values more similar[-1, 1]Normalized vectors
L2 (Euclidean Distance)Lower values more similar[0, ∞)Image features
JACCARDLower values more similar[0, 1]Binary vectors
HAMMINGLower values more similar[0, dim]Binary vectors

⚠️ Key Point: The meaning of distance field depends on metric_type. For COSINE/IP, higher distance is better; for L2, lower distance is better.

II. Prerequisite: Collection Loading

Collections must be loaded into memory before searching, otherwise errors will occur.

def ensure_collection_loaded():
    """Ensure collection is loaded into memory, must be called before search"""
    load_state = client.get_load_state(collection_name=collection_name)

    # state is enum type (LoadState), str(LoadState.Loaded) == 'Loaded'
    if str(load_state.get('state')) != 'Loaded':
        client.load_collection(
            collection_name=collection_name,
            # load_fields optional: Load only needed fields to save memory
            # If not specified, loads all fields
            load_fields=["book_id", "title", "content_vector", "keyword_vector"]
        )

load_fields trade-offs:

  • Specify load_fields: Save memory, but output_fields can only contain loaded fields
  • Not specify: Load all fields, flexible but high memory usage
  • Vector fields (used for search) must be included in load_fields

III. Single Vector Search

Most basic search method, input one query vector, return top-K most similar results.

def single_vector_search():
    """
    Single vector search: One query, return K most similar records

    search() return value structure:
    res[0]          → Results list for first query vector
    res[0][i]       → i-th hit result
    res[0][i]['id'] → Hit entity's primary key
    res[0][i]['distance'] → Distance from query vector (similarity)
    res[0][i]['entity']   → Hit entity's field values (determined by output_fields)
    """
    # Generate 256-dimensional query vector (actually comes from embedding model in real scenarios)
    query_vector = [float(random.uniform(-1, 1)) for _ in range(256)]

    result = client.search(
        collection_name=collection_name,
        data=[query_vector],           # Single vector passes 1D list, multiple vectors pass 2D list
        anns_field="content_vector",   # Specify which vector field to search on
        limit=3,                       # Return top 3 most similar
        search_params={"metric_type": "COSINE"},
        output_fields=["book_id", "title", "rating"]  # Scalar fields to return
    )

    # result is 2D list: result[query_vector_index][hit_result_index]
    for i, hit in enumerate(result[0]):
        print(f"{i+1}. ID: {hit['entity']['book_id']}, "
              f"Title: {hit['entity']['title']}, "
              f"Similarity: {hit['distance']:.4f}")

Common misconception:

  • limit maximum value limited by collection configuration, default upper limit 16384

IV. Batch Vector Search

Input multiple query vectors in one request, Milvus processes in parallel, more efficient than multiple individual searches.

def bulk_vector_search():
    """
    Batch search: Process multiple query vectors in one request

    Application scenarios:
    - Batch recommendations (recommend to multiple users simultaneously)
    - Batch similarity calculations
    - Reduce network round trips
    """
    # Prepare 3 query vectors
    query_vectors = [
        [float(random.uniform(-1, 1)) for _ in range(256)]
        for _ in range(3)
    ]

    result = client.search(
        collection_name=collection_name,
        data=query_vectors,   # Pass multiple vectors
        anns_field="content_vector",
        limit=2,
        search_params={"metric_type": "COSINE"},
        output_fields=["book_id", "title"]
    )

    # result is 2D list, result[i] corresponds to i-th query vector's results
    print(f"Total {len(result)} result groups")  # → 3
    for i, hits in enumerate(result):
        print(f"\nResults for query vector {i+1}:")
        for j, hit in enumerate(hits):
            print(f"  {j+1}. {hit['entity']['title']}, Similarity: {hit['distance']:.4f}")

💡 Performance tip: Total time for batch search is usually much less than sum of multiple individual searches, recommended when multiple searches needed.

V. Filtered Search

Add scalar field filter conditions on top of vector search to narrow search scope.

5.1 Two Filtering Method Comparison

Standard Filtering:
  ┌─────────────┐    ┌──────────────┐    ┌──────────┐
  │  Full Data  │ → │ Scalar Filter │ → │ ANN Search │ → top-K
  └─────────────┘    └──────────────┘    └──────────┘
  Filter first, then search. Good performance when filter conditions simple.

Iterative Filtering:
  ┌─────────────┐    ┌──────────────┐    ┌──────────────┐
  │  Full Data  │ → │ ANN Iterative │ → │ Per-item     │ → top-K
  └─────────────┘    └─── Search ────┘    └── Filtering ──┘
  Search first, then filter item by item. Good performance when filter conditions complex.
Comparison ItemStandard FilteringIterative Filtering
Filtering TimingBefore ANN searchAfter ANN search
Application ScenariosLarge remaining data after filteringSmall remaining data after filtering
Enable MethodDefault, no extra parametersAdd "hints": "iterative_filter" in search_params
Potential IssuesHigh latency with complex filter conditionsPoor performance when very few qualifying data need large iterations

5.2 Standard Filtering

def filtered_vector_search():
    """Standard filtered search: First filter scalar fields, then ANN search in filtered results"""
    query_vector = [float(random.uniform(-1, 1)) for _ in range(256)]

    result = client.search(
        collection_name=collection_name,
        data=[query_vector],
        anns_field="content_vector",
        limit=3,
        search_params={"metric_type": "COSINE"},
        # filter supports operators: ==, !=, >, >=, <, <=, and, or, not, like, in, array_contains, etc.
        filter="is_available == True and rating >= 4.0",
        output_fields=["book_id", "title", "is_available", "rating"]
    )

    for i, hit in enumerate(result[0]):
        entity = hit['entity']
        print(f"{i+1}. {entity['title']}, "
              f"Available: {entity['is_available']}, "
              f"Rating: {entity['rating']}, "
              f"Similarity: {hit['distance']:.4f}")

Common Filter Expressions:

# Boolean fields
filter="is_available == True"

# Numeric ranges
filter="rating >= 4.5"
filter="price > 50 and price < 200"

# String fuzzy matching (% wildcard)
filter='title like "%Python%"'

# IN list
filter="book_id in [1, 2, 3]"

# Compound conditions
filter="is_available == True and rating >= 4.0 and price < 100.0"

# JSON field access
filter='details["author"] == "Zhang San"'

# ARRAY field contains
filter='array_contains(tags, "programming")'

5.3 Iterative Filtering

def iterative_filtered_search():
    """
    Iterative filtered search: Suitable for complex filter conditions

    Principle: ANN search iteratively returns candidate results, each candidate individually filtered,
    until limit qualifying results collected.

    Advantages: Reduce scalar filtering workload (only filter ANN candidates, not full data)
    Disadvantages: If very few qualifying data, need large iterations, performance degrades
    """
    query_vector = [float(random.uniform(-1, 1)) for _ in range(256)]

    result = client.search(
        collection_name=collection_name,
        data=[query_vector],
        anns_field="content_vector",
        limit=3,
        search_params={
            "metric_type": "COSINE",
            "hints": "iterative_filter"   # ← Only difference, enable iterative filtering
        },
        # Complex filter conditions (multi-field, multi-condition) better suited for iterative filtering
        filter="is_available == True and rating >= 4.0 and price < 100.0",
        output_fields=["book_id", "title", "is_available", "rating", "price"]
    )

    for i, hit in enumerate(result[0]):
        entity = hit['entity']
        print(f"{i+1}. {entity['title']}, "
              f"Rating: {entity['rating']}, "
              f"Price: ¥{entity.get('price', 'N/A')}, "
              f"Similarity: {hit['distance']:.4f}")

5.4 How to Choose Filtering Method

The core difference between the two methods is proportion of remaining data after filtering:

  • Large remaining data after filtering (most data passes filter) → Standard filtering
  • Small remaining data after filtering (few data passes filter) → Iterative filtering

💡 Practical suggestion: When uncertain, actually measure both methods’ execution time like in filter-search-demo.py’s performance_comparison(), choose based on data distribution.

VI. Pagination Search

When needing results exceeding limit upper bound, or implementing pagination, use offset parameter.

def pagination_search():
    """
    Pagination search: Skip first N results via offset, implement pagination

    limit + offset sum cannot exceed 16384 (Milvus default limit)
    """
    query_vector = [float(random.uniform(-1, 1)) for _ in range(256)]

    page_size = 2  # 2 records per page

    for page in range(3):  # Get first 3 pages
        offset = page * page_size

        result = client.search(
            collection_name=collection_name,
            data=[query_vector],
            anns_field="content_vector",
            limit=page_size,
            search_params={
                "metric_type": "COSINE",
                "offset": offset    # ← offset in search_params
            },
            output_fields=["book_id", "title"]
        )

        print(f"\nPage {page+1} (offset={offset}):")
        for i, hit in enumerate(result[0]):
            print(f"  {i+1}. {hit['entity']['title']}")

Pagination Notes:

ConsiderationDescription
offset positionCan be in search_params dictionary, or as top-level keyword parameter of search()
Upper limitlimit + offset <= 16384, will error if exceeded
Consistency issuesResults may be inconsistent across pages if data written during pagination
PerformanceLarger offset, worse performance (need to skip first N results)

⚠️ Large data pagination: For traversing large amounts of data (like exports), recommend using Search Iterator (search iterator), better performance, covered in later Part.

VII. output_fields Usage Tips

# Return specified fields
output_fields=["book_id", "title", "rating"]

# Return all scalar fields (excluding vector fields)
output_fields=["*"]

# Return vector fields (large data volume, use as needed)
output_fields=["book_id", "content_vector"]

# Access JSON field sub-properties
output_fields=["details"]

VIII. Search Result Structure Quick Reference

result = client.search(...)

# result structure
result                    # List[List[Hit]], outer layer corresponds to each query vector
result[0]                 # All hit results for first query vector
result[0][0]              # First hit result (most similar)

hit = result[0][0]
hit['id']                 # Entity primary key (quick access)
hit['distance']           # Distance/similarity from query vector
hit['entity']             # Dictionary containing fields specified by output_fields
hit['entity']['title']    # Access specific field values
